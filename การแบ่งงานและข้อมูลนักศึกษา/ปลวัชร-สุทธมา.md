## 1. Custom Hook `useLocalStorage`

### หน้าที่และหลักการทำงาน

`useLocalStorage` รวมการอ่าน เขียน และตรวจข้อมูล Local Storage ไว้ในจุดเดียว รับพารามิเตอร์ `key`, `initialValue` และ `validator` แล้วคืนค่า `[storedValue, updateStoredValue, isHydrated]`

Effect แรกอ่านข้อมูลจากเบราว์เซอร์และใช้ `JSON.parse` ภายใน `try/catch` หากข้อมูลไม่ถูกต้องจะกลับไปใช้ค่าเริ่มต้น:

```js
useEffect(() => {
  try {
    const savedValue = window.localStorage.getItem(key)

    if (savedValue !== null) {
      const parsedValue = JSON.parse(savedValue)
      setStoredValue(validator(parsedValue) ? parsedValue : initialValue)
    }
  } catch {
    setStoredValue(initialValue)
  } finally {
    setIsHydrated(true)
  }
}, [initialValue, key, validator])
```

Effect ที่สองบันทึกข้อมูลเฉพาะหลังโหลดค่าเดิมเสร็จแล้ว เพื่อป้องกัน Array ว่างเริ่มต้นเขียนทับตะกร้าเดิม

```js
useEffect(() => {
  if (!isHydrated) return

  try {
    window.localStorage.setItem(key, JSON.stringify(storedValue))
  } catch {
    // เว็บยังใช้งานได้เมื่อ Storage ไม่พร้อมใช้งาน
  }
}, [isHydrated, key, storedValue])
```

### เหตุผลที่เลือกใช้

Local Storage เป็นระบบภายนอก React และเป็น Side Effect จึงเหมาะกับ `useEffect` การแยกเป็น Custom Hook ทำให้ `CartContext` ไม่ต้องเขียนโค้ด Browser API ซ้ำและนำ Hook ไปใช้กับข้อมูลอื่นในอนาคตได้

### ข้อดี

- จัดการ JSON และ Error ไว้ในที่เดียว
- ข้อมูลเสียไม่ทำให้หน้าเว็บพัง
- ป้องกันการเขียนทับข้อมูลก่อน Hydration เสร็จ
- ใช้ซ้ำกับ Storage Key อื่นได้

### ข้อจำกัดและข้อควรระวัง

- Local Storage ใช้ได้เฉพาะฝั่ง Browser และเก็บข้อมูลได้จำกัด
- ข้อมูลไม่ซิงก์ข้ามอุปกรณ์และไม่เหมาะกับข้อมูลลับ
- Dependency ของ Effect ต้องครบเพื่อป้องกันข้อมูลล้าสมัย
- ต้องตรวจข้อมูลจาก Storage ก่อนนำมาใช้เสมอ

## 2. ฟังก์ชันตรวจสอบ `isValidCart`

### หน้าที่และหลักการทำงาน

ก่อนนำข้อมูลจาก Local Storage เข้า Reducer ระบบตรวจว่าเป็น Array และแต่ละรายการมี ID, ชื่อ, ราคา, หมวดหมู่, รูปภาพ และจำนวนที่ถูกต้อง รวมทั้งตรวจว่า ID ไม่ซ้ำ

```js
const isValidItem =
  Number.isInteger(item?.id) &&
  typeof item?.name === 'string' &&
  typeof item?.price === 'number' &&
  Number.isFinite(item.price) &&
  item.price >= 0 &&
  Number.isInteger(item?.quantity) &&
  item.quantity >= 1 &&
  !ids.has(item.id)
```

ข้อดีคือป้องกัน State ที่ผิดรูปแบบและค่าติดลบ ข้อจำกัดคือหากโครงสร้างสินค้าเปลี่ยนต้องปรับ Validator ให้ตรงกัน

## 3. `useCallback` สำหรับฟังก์ชันตะกร้า

### หน้าที่และหลักการทำงาน

ฟังก์ชัน `addItem`, `removeItem`, `increaseQuantity`, `decreaseQuantity` และ `clearCart` ถูกห่อด้วย `useCallback` ก่อนส่งผ่าน Context:

```js
const decreaseQuantity = useCallback((productId) => {
  dispatch({
    type: CART_ACTIONS.DECREASE_QUANTITY,
    payload: productId,
  })
}, [])
```

### เหตุผลที่เลือกใช้

ฟังก์ชันเหล่านี้ถูกใช้งานโดย Component ลูกที่ครอบด้วย `React.memo` การรักษา Reference ของ Callback ให้คงที่ช่วยให้ Memoization มีผลและลดการ Render จากฟังก์ชัน Object ใหม่

### ข้อดี

- ทำงานร่วมกับ `React.memo` ได้มีประสิทธิภาพขึ้น
- Context Value เปลี่ยนน้อยลงจาก Reference ที่ไม่จำเป็น
- ทำให้ API จัดการตะกร้ามีรูปแบบคงที่

### ข้อจำกัดและข้อควรระวัง

- Dependency ต้องครบ มิฉะนั้น Callback อาจใช้ค่าเก่า
- ไม่ควรใช้ `useCallback` กับทุกฟังก์ชัน เพราะเพิ่มความซับซ้อนโดยไม่มีประโยชน์เสมอไป
- การใช้ `useCallback` ไม่ได้ป้องกัน Render หากข้อมูลอื่นใน Context เปลี่ยน

## 4. Custom Hook `useCart`

`useCart` เป็นทางเข้าใช้งาน `CartContext` สำหรับทุก Component และตรวจว่าถูกเรียกภายใน `CartProvider` หากไม่มี Provider จะ Throw Error ที่สื่อความหมาย ช่วยตรวจข้อผิดพลาดของโครงสร้าง Component ได้เร็ว

## 5. การจัดการรูปภาพสำรอง

`ProductCard` ใช้ `useState` เก็บสถานะ `imageFailed` เมื่อรูปจาก URL ภายนอกโหลดไม่ได้ `onError` จะเปลี่ยน State และแสดงสัญลักษณ์ตามหมวดหมู่แทน ทำให้การ์ดยังมีภาพสื่อความหมายและ Layout ไม่เสีย

```jsx
<img
  src={product.image}
  alt={product.name}
  loading="lazy"
  onError={() => setImageFailed(true)}
/>
```

ข้อดีคือรองรับปัญหาเครือข่ายได้ ส่วนข้อจำกัดคือภาพสำรองเป็นสัญลักษณ์ทั่วไป ไม่ใช่ภาพสินค้าจริง

## 6. การตรวจสอบ Responsive

`src/index.css` ใช้ CSS Grid, Flexbox และ Media Queries โดยมีพฤติกรรมหลักดังนี้:

- จอใหญ่: เมนู 3 คอลัมน์และตะกร้าอยู่ด้านขวา
- ไม่เกิน 1180px: เมนูเหลือ 2 คอลัมน์
- ไม่เกิน 920px: ตะกร้าย้ายลงด้านล่าง
- ไม่เกิน 640px: เมนู 1 คอลัมน์ ปุ่มมีขนาดกดง่าย
- ไม่เกิน 410px: การ์ดสินค้าเปลี่ยนเป็นแนวตั้งเต็มความกว้าง

ข้อดีคือรองรับ 375px, 768px, 1024px และ 1440px โดยไม่มี Horizontal Scroll ข้อควรระวังคือเมื่อเพิ่มเนื้อหาหรือ Component ใหม่ต้องทดสอบ Breakpoint ทั้งหมดอีกครั้ง
